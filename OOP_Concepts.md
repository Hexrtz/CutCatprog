# Object-Oriented Programming (OOP) Concepts ใน T-Rex Runner

จากการตรวจสอบซอร์สโค้ดในโปรเจกต์ T-Rex Runner พบว่ามีการใช้หลักการ OOP ครบทั้ง 8 หัวข้อตามที่ระบุไว้ครับ มีรายละเอียดและตัวอย่างในโค้ดดังนี้:

## 1. Encapsulation (การห่อหุ้มข้อมูล)
มีการซ่อนข้อมูล (Data Hiding) ไว้ภายในคลาสโดยบรรจุใส่ตัวแปรลักษณะ `private` หรือ `protected` และเปิดให้เข้าถึงผ่าน `public` methods เพื่อป้องกันมิให้ส่วนอื่นมาแก้ไขตัวแปรโดยตรง
- **ตัวอย่าง**: ในคลาส `Character` มีการซ่อนสถานะ `private double velocityY` และ `private boolean isJumping` ซึ่งจะถูกเปลี่ยนแปลงค่าได้ผ่านเมธอดสาธารณะอย่าง `jump()` หรือ `update()` เท่านั้น
- **ตัวอย่าง**: ในคลาส `GameScreen` มีการประกาศ `private int score` และ `private int state` เพื่อจัดการสถานะของเกมภายในคลาสเดียว ไม่ให้คลาสอื่นแก้คะแนนหรือสถานะมั่วซั่วได้

## 2. Association (ความสัมพันธ์ระหว่างคลาส)
เป็นการประกอบกันระหว่างคลาส (Has-a relationship) ที่ออบเจกต์หนึ่งมีอีกออบเจกต์หนึ่งเป็นส่วนประกอบในคลาสตัวเอง
- **ตัวอย่าง**: โปรเจกต์นี้มีคอมโพเนนต์มากมายใน `GameScreen` แบบ Association โดยตัวหน้าจอเกมมีการถือออบเจกต์ของคลาสต่างๆ เช่น `private Character player;`, `private List<Obstacle> obstacles;`, `private Background sky;` และ `private AudioPlayer audioPlayer;` ซึ่งเป็นส่วนหนึ่งของการทำงานของเกมทั้งหมด

## 3. Inheritance (การสืบทอดคุณสมบัติ)
มีการสืบทอดลักษณะและโครงสร้างจากคลาสแม่สู่คลาสลูก (Is-a relationship) ด้วยคีย์เวิร์ด `extends` เพื่อให้สามารถใช้โค้ดซ้ำและต่อยอดความสามารถได้
- **ตัวอย่าง**: คลาส `Background`, `Character`, และ `Obstacle` ล้วนสืบทอดพันธุกรรมโครงสร้างตำแหน่งและรูปภาพมาจากคลาสแม่อย่าง `GameObject`
- **ตัวอย่าง**: คลาส `Rabbit` (กระต่ายศัตรู) ถูกสืบทอดมาจากคลาสแม่อย่าง `Obstacle` อีกทอดหนึ่งเพื่อให้มีเรื่องของสปีดเพิ่มเข้ามา
- **ตัวอย่าง**: คลาส `GameScreen` มีสืบทอดคอมโพเนนต์ UI พื้นฐานของ Java อย่าง `JPanel`

## 4. Polymorphism (พหุสัณฐาน)
ออบเจกต์มีลักษณะที่สามารถแสดงพฤติกรรมได้หลายรูปแบบ หรือสามารถจัดการลูกๆ ในรูปแบบของแม่เดียวกันได้
- **ตัวอย่าง (Subtyping)**: มีการประกาศลิสต์แบบ `List<Obstacle> obstacles = new ArrayList<>();` แต่ตอนนำข้อมูลเข้า สามารถเพิ่มตัวกระต่ายผ่านการเรียก `obstacles.add(new Rabbit(...))` ซึ่งส่งออบเจกต์ลูกไปเก็บในลิสต์ของคลาสแม่ได้
- **ตัวอย่าง (Method Overriding / Dynamic Binding)**: เมื่อมีการใช้งานในลูป `for (Obstacle obs : obstacles)` และเรียกใช้งาน `obs.update()` หรือ `obs.draw(g2d)` รูปแบบพฤติกรรมการทำงานของพวกมันจะเป็นพฤติกรรมของคลาสลูก (อย่างเช่นการทำงานของ `Rabbit`) แบบอัตโนมัติ 

## 5. Overloading (การโอเวอร์โหลดเมธอด)
เป็นคุณสมบัติในระดับเดียวกันที่คลาสหนึ่งสามารถมีเมธอดชื่อเดียวกันหลายๆ เมธอดได้ แต่รับพารามิเตอร์ไม่เหมือนกัน (Signature หรือรับข้อมูลแตกต่างกัน)
- **ตัวอย่าง Constructor**: ในคลาส `Character` มีฟีเจอร์นี้ คือ
  - `public Character(double x, double y, int width, int height, Image image)` สำหรับสร้างตัวละครแบบระบุขนาด
  - `public Character(double x, double y, Image image)` สำหรับสร้างตัวละครโดยขยายเป็นค่าขนาดตั้งต้นแบบขนาดใหญ่
- **ตัวอย่าง Method**:
  - `public void jump()` (ใช้แรงกระโดดเริ่มต้น -12)
  - `public void jump(double customForce)` (รับค่าแรงกระโดดที่กำหนดเองได้แบบ parameter)

## 6. Overriding (การแทนที่เมธอด)
การกำหนดการทำงานใหม่ให้กับเมธอดในคลาสลูก ที่เกิดการสืบทอดมาจากคลาสแม่หรือ Interface (มักถูกกำกับด้วยคำว่า `@Override`) 
- **ตัวอย่าง**: เมธอดนามธรรม `update()` ใน `GameObject` ถูก Override เนื้อหาการทำงานในคลาสลูกต่างๆ เช่น `Background`, `Character`, และ `Rabbit` เพื่อให้เคลื่อนที่เป็นรูปแบบของตัวเอง
- **ตัวอย่าง**: เมธอด `getBounds()` ตอนตรวจจับการชน ใน `Character` กับ `Obstacle` ได้ถูกเขียน Overide ทับเพื่อปรับขนาดขนาด Hitbox (กล่องการชน) ให้เหมาะสมกับภาพ
- **ตัวอย่าง**: คลาส `GameScreen` นำเมธอดมาจาก `Runnable` และ `JPanel` มีการ override เช่น `run()` และ `paintComponent(Graphics g)`

## 7. Interface Class (อินเตอร์เฟซ)
เป็นคลาสต้นแบบข้อตกลงและสัญญา (Contract) ที่คลาสต่างๆ ที่นับถือมันจะต้องนำไปสร้างการทำงานทั้งหมด 
- **ตัวอย่าง (`Renderable`)**: มีข้อกำหนดหนึ่งคือ `void draw(Graphics2D g2d);` ซึ่ง `GameObject` ทุกตัวก็นำข้อเสนอนี้ไปยอมรับผ่านคีย์เวิร์ด `implements`
- **ตัวอย่าง (`Collidable`)**: มีข้อกำหนดให้สร้าง `Rectangle getBounds();` (และแถมมี default method ให้อย่าง `isColliding()`) ถูกนำไปร่วมใช้งานโดยกลไกที่สามารถชนกันได้เช่น `Character` และ `Obstacle`
- **ตัวอย่าง (`Runnable`)**: นำมาใช้เพื่อให้หน้าจอควบคุม Thread และรันเกมลูปต่อเนื่อง

## 8. Abstract Class (คลาสนามธรรม)
เป็นคลาสต้นแบบที่ไม่สามารถถูกสร้างเป็นอ็อบเจกต์ได้ด้วยตัวเองโดยตรง แต่มีไว้เพื่อระบุโครงสร้างการทำงานบางอย่างเพื่อให้คลาสลูกนำไปสานต่อ
- **ตัวอย่าง (`GameObject`)**: นิยามให้เป็น `public abstract class GameObject` โดยสร้างกฎนามธรรมตัวนึงขึ้นมาขื่อ `public abstract void update();` เพื่อกดบังคับให้คลาสลูกต่างๆ ต้องไปต่อเติมเอาเอง
- **ตัวอย่าง (`Obstacle`)**: สิ่งกีดขวางเป็นต้นแบบของศัตรูทุกตัวในตระกูล ได้รับการนิยามเป็น `public abstract class Obstacle extends GameObject` และมีการรวม logic ความเร็ว (Speed) ไว้อย่างชัดเจน
